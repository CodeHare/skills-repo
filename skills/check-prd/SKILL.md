# check-prd - B端PRD质量检查

对 B端产品的 PRD 和系统设计方案做全面、系统化的质量检查，覆盖 14 个维度。

> 作者：杨堃，《决胜B端》作者，B端产品专家
> 基于：《决胜B端（第二版）》《决胜体验设计》《决胜B端PRD模板 v2.0》

## 输入

- 用户提供的待检查 PRD / 系统设计文档内容（直接粘贴或上传文件）
- 或通过 `$ARGUMENTS` 传入的文件路径

如有参数传入，优先作为输入源：

$ARGUMENTS

## 执行流程

严格按以下顺序执行，不可跳过或调换步骤。**核心原则：逐维度即时输出**——每完成一个维度检查，立即输出该维度的详细分析，然后再继续下一个维度。禁止把所有维度攒到最后统一输出。

### 第零步：产品定型

在开始任何维度检查前，先阅读文档并判断以下两个维度：

**A. 商业属性：**
- 自研内部系统：企业内部使用，不对外销售
- 商业化产品：对外销售或运营，存在付费客户

**B. 功能类型：**
- 业务型管理软件：支撑核心业务流程运转
- 工具型软件：提供独立功能提升效率
- 交易型平台：撮合多方完成交易
- 基础服务型：提供底层能力供其他系统调用

**根据定型结论，以下维度按条件适用：**

| 维度 | 适用条件 | 不适用时处理方式 |
| --- | --- | --- |
| 07-数据建模（ER图） | 仅适用于业务型管理软件和交易型平台 | 标注"不适用"，仅检查关键数据结构 |
| 10-商业分析 | 仅适用于商业化产品 | 仅执行投入产出分析 |
| 03 维度中的竞品分析 | 仅适用于商业化产品 | 改为同类系统参考 |
| 06-企业架构层 6.5-6.9 | 仅适用于 0-1 新系统设计或系统级规划 PRD | 只做 6.1-6.4 |
| R8-多租户风险项 | 仅适用于商业化 SaaS 产品 | 标注"不适用" |
| 13-AI 功能 | 仅在文档涉及 AI 功能时评估 | 标注"不适用" |

**输出产品定型结论**，格式：

> 📋 **产品定型**：{商业属性} × {功能类型}
> 据此，以下维度调整适用性：[列出不适用或调整侧重的维度]

## 输出规范

### 每个维度的输出格式

```md
## 维度[编号] - [名称] ｜ 评级：[优秀 / 合格 / 待改进 / 严重缺失]

### 具体发现

**发现 1：[问题标题]** [P0/P1/P2/P3]
- PRD定位：第X节/[功能名称]
- 问题描述：[具体说明问题，不要泛泛而谈]
- 改进示例：[给出可立刻执行的改法]

### 隐性问题推断
结合产品类型和业务场景，列出 PRD 没写但按道理必须考虑的问题。
```

### 最低输出要求

1. 每个维度至少输出 3 条具体发现，若确实无问题，也要逐条说明理由
2. 每条发现必须引用 PRD 中的具体章节、流程、功能名或界面名
3. 每条发现必须给出具体可执行的改进示例
4. 维度 09 必须逐个检查文档中出现的界面、弹窗、表单、操作，并指出缺失的交互说明

### 汇总报告结构

所有维度检查完毕后，按 `references/report-template.md` 模板输出：

1. **产品定型说明**
2. **各维度发现摘要**（含适用性标注）
3. **重大风险项**（R1-R8，含存在/不存在/不适用判断）
4. **问题清单**（按 P0/P1/P2/P3 分级）
5. **亮点记录**
6. **Top 10 改进建议**
7. **综合建议**

不做通过/不通过的判定，最终如何推进由 PRD 作者和团队根据实际情况决定。

## 工作风格

- 目标是给出专业、具体、可操作的改进建议，帮助 PRD 作者提升文档质量
- 每个问题必须引用 PRD 中的具体章节或功能名，不能泛泛而谈
- 每个问题必须给出具体可执行的改进示例
- 区分"必须改"和"建议改"，给 PRD 作者明确的行动指引
- 检查不只是找问题，优秀的设计也值得记录和学习
- 沟通时先说亮点再说问题，建立积极的沟通氛围
