# 第10章 功能需求 — 生成指引

> 这是 PRD 中最核心、最大的章节。

## 章节目标

以结构化方式详尽描述产品的功能设计，包括产品框架、功能详解和异常处理。

> 💡 方法论基础：
> - 产品框架：《决胜B端》自顶向下设计
> - 数据建模：《决胜B端》ER建模三步法（找实体→梳关系→定属性）
> - 流程设计：UML 建模（泳道图 + 状态机）
> - 交互设计：四大设计原则（有用、高效、容错、启发）
> - 规则描述：五种规则类型（事实、约束、触发条件、推论、计算）

## 生成结构

### 10.1 产品框架概述

所有图表使用 Mermaid 语法。

#### 10.1.1 应用架构图

使用 `graph TB`，用 `subgraph` 分层：
- 分为用户层、接入层、业务服务层、数据层、外部系统 五层
- 外部系统用虚线箭头 `-.->` 表示
- 每个模块节点用 `<br/>` 换行

#### 10.1.2 数据模型图

使用 Mermaid `erDiagram` 语法。

关系符号：
- `||--||` 一对一
- `||--|{` 一对多
- `}|--|{` 多对多
- `||--o|` 零或一对一

**生成规则：**
- 业务型/交易型产品：ER图必须完整
- 每个实体列出关键属性（PK/FK/状态/核心业务字段）
- ER图后面附实体说明表格

#### 10.1.3 核心业务流程图

使用 Mermaid `flowchart TD` 语法：
- `([...])` 开始/结束节点
- `{...}` 判断/分支节点
- `[...]` 普通步骤
- 关键状态着色（蓝色=入口、绿色=成功、红色=失败、橙色=等待）
- 涉及多角色时用 `subgraph` 划分泳道

#### 10.1.4 状态机图

使用 Mermaid `stateDiagram-v2` 语法：
- 为每个有状态流转的核心实体画一张状态机
- 包含正常路径和异常路径
- 用 `note` 标注关键约束
- 状态机图后附状态转换明细表

### 10.3 异常情况处理方案

| 异常类型 | 异常场景 | 处理方案 |
| --- | --- | --- |
| 网络异常 | 提交操作时断网 | 本地暂存，恢复后自动重试 |
| 并发冲突 | 多人同时编辑同一记录 | 乐观锁 + 冲突提示 |
| 数据异常 | 外部系统返回错误数据 | 校验拦截 + 错误日志 |
| 误操作 | 误删除重要数据 | 软删除 + 回收站 |
| 业务异常 | 审批超时无人处理 | 超时提醒 → 升级提醒 → 自动转派 |

### AI 功能设计（如适用）

如果产品包含 AI 功能，额外加入：
- **确定性-容错性四象限分析**
- **六脉神剑交互模式**：封装API / CUI嵌入GUI / Chat / Copilot / 辅助填充 / 后台自动化
- **降级方案**
- **监控指标**：准确率、使用率、弃用率

## 不同产品类型的侧重

| 功能类型 | 10.1 侧重 | 10.2 侧重 | 10.3 侧重 |
| --- | --- | --- | --- |
| 业务型 | ER图+流程图+状态机必须完整 | 角色交互和审批流程 | 审批异常、数据一致性 |
| 工具型 | 简化模型，强调使用流程 | 操作效率和快捷方式 | 数据丢失防护、撤销操作 |
| 交易型 | ER图+交易流程图+状态机 | 交易闭环各环节规则 | 支付异常、库存并发、对账 |
| 基础服务型 | API结构+调用流程 | 接口规范和参数说明 | 服务降级、限流、重试 |

## 质量标准
- 10.1 至少包含系统框架图和功能清单
- 业务型/交易型产品有完整ER模型
- 有状态流转的实体都有状态机
- 每个模块按"流程→交互→规则"三段式
- 10.3 至少覆盖5类异常场景
